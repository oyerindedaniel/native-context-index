# The on-disk schema

<PageLede>
  The index is one SQLite file. Get the canonical DDL with `nci sql --schema`. The tables below are the public surface that `query` and `sql` read from — column names match what the live database exposes.
</PageLede>

## The authoritative source

<TerminalRoot>
  <TerminalCommand>nci sql --schema</TerminalCommand>
</TerminalRoot>

That command prints the live `CREATE TABLE` statements straight from the open database. Everything below summarises those statements; if anything here disagrees with `--schema`, **trust `--schema`**.

## Inspecting it

<TerminalRoot>
  <TerminalCommand>nci sql --format jsonl -c "SELECT name, version FROM packages ORDER BY name LIMIT 5"</TerminalCommand>
</TerminalRoot>

`nci sql` only accepts read-only statements. Mutating SQL (`INSERT`, `UPDATE`, `DELETE`, `DROP`) is rejected before it ever touches the database.

<CalloutRoot variant="info">
  <CalloutTitle>Two id shapes on `symbols`.</CalloutTitle>
  <CalloutBody>
    <code>symbols.symbol_id</code> is the integer row id used by the dependency tables (<code>symbol_dependencies.from_symbol_id</code>, etc.). <code>symbols.id</code> is the stable string handle used by <code>nci query show</code> / <code>snippet</code> / <code>overloads</code> and by <code>symbol_dependencies.to_symbol_id_text</code>. Pick the right one for the join.
  </CalloutBody>
</CalloutRoot>

## Migrations and journaling

The CLI applies pending migrations on `nci db init` and on the first `nci index`. Use `nci db status --check` for a quick `PRAGMA quick_check`, `--deep` for the full integrity check, and `nci db wal-checkpoint` to truncate the WAL after a long indexing run.
