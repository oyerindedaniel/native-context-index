# Exit codes

<PageLede>
  The CLI uses two exit code spaces: clap's defaults for argument parsing, and one NCI-specific code for &ldquo;the symbol id you asked about does not exist&rdquo;. Anything else surfaces as a generic failure.
</PageLede>

<CalloutRoot variant="warning">
  <CalloutTitle>Don&rsquo;t branch on `2` for usage errors alone.</CalloutTitle>
  <CalloutBody>
    Because clap and `EXIT_QUERY_NOT_FOUND` both exit with `2`, scripts that distinguish &ldquo;not found&rdquo; from &ldquo;bad command line&rdquo; should read the JSON envelope on stdout (`--format json`) instead of relying on the code alone. The not-found path is the only one that returns a structured <code>data</code> field with a `null`-shaped payload (e.g. `data.symbol: null`).
  </CalloutBody>
</CalloutRoot>

## Reading the JSON envelope

For commands that support `--format json`, failures print `{ "ok": false, "error": "<string>" }` to stdout — `error` is a **single human-readable string**, not an object (see `json_err` in `packages/nci-engine/src/cli.rs`). There is no separate `kind` / `message` split in the binary today.

For the success envelopes and `data` shapes used by `nci query`, see [JSON output schemas](/docs/reference/json-schemas).
