## Field deep-dives

### project_root

The directory whose `package.json` and `node_modules` are scanned. Relative paths resolve against the file containing the config — not against `cwd`. If you run `nci index` from a workspace, the file&rsquo;s `project_root` still wins unless `--project-root` is passed.

### package_scope

Either an array of sections or the sentinel.

```json
{
  "package_scope": ["dependencies"]
}
```

Indexes packages listed in the consumer manifest&rsquo;s `dependencies`. Add `"dev_dependencies"` to include build tools.

```json
{
  "package_scope": "all_installed"
}
```

Disables the manifest gate. Every installed package is indexed regardless of how it got there. Use sparingly — large transitive trees are not free.

### max_hops

The depth of re-export resolution.

| Value | Behaviour                                      |
| ----- | ---------------------------------------------- |
| `0`   | Index entry-file declarations only             |
| `1`   | Follow one level (entry → re-exported file)    |
| `10`  | Default — covers practically every real lib    |
| `-1`  | Unlimited — only useful for diagnostics        |

### workspaces

Glob list, relative to `project_root`. Each entry contributes its own `node_modules` install root.

```json
{
  "workspaces": ["apps/*", "packages/*"]
}
```

### dependency_stub_packages

Names a dependency that should be persisted as `npm::…` stub edges instead of being crawled. Whenever a crawled file imports a listed name (or a subpath of it), NCI records the edge as `npm::<specifier>::<member>` and skips its `.d.ts`. Use it for shared validators or heavy SDKs (`zod`, `@aws-sdk/client-s3`, `googleapis`) that you reference but do not need walked.
